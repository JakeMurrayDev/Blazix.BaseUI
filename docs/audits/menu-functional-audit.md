# Menu Functional Audit

Date: 2026-06-30

## Scope

Audited and repaired the Blazix.BaseUI **Menu** component port against the vendored React Base UI source at `.base-ui` commit `748f4228d`. The Menu surface is the largest popup family in the library — 19 public parts plus `Separator` and the `createHandle`/`Handle` detached-trigger API — and is the only popup component that exercises list-navigation, typeahead, submenu hover (safePolygon), checkbox/radio item families, and group-label wiring.

Primary source surfaces inspected:

- `.base-ui/packages/react/src/menu/**` (root, store/handle, trigger, portal, positioner, popup, arrow, backdrop, item, link-item, checkbox-item(+indicator), radio-group, radio-item(+indicator), group, group-label, submenu-root, submenu-trigger, viewport, utils)
- `.base-ui/packages/react/src/floating-ui-react/**` (`useListNavigation`, `useTypeahead`, `useHover*`, `safePolygon`, `FloatingFocusManager`, `useHoverInteractionSharedState`, `utils/composite`)
- `.base-ui/packages/react/src/composite/**`, `.base-ui/packages/react/src/utils/popups/**`, `utils/popupStateMapping.ts`
- `.base-ui/docs/src/app/(docs)/react/components/menu/**`
- `src/Blazix.BaseUI/Menu/**`, `src/Blazix.BaseUI/wwwroot/blazix-baseui-menu.js` (+ `.min.js`), `blazix-baseui-floating.js`, `blazix-baseui-composite.js`
- `src/Blazix.BaseUI/Utilities/FloatingFocusManager/**`

## Method

The audit was executed with four parallel read-only sub-agents — (1) exhaustive React Menu source catalog, (2) exhaustive Blazor Menu port catalog, (3) upstream-delta `git show` analysis of every recent `[menu]`/`[composite]`/`[popups]`/`[floating]` commit since 2026-02, and (4) a shared JS-infrastructure sweep mapping `blazix-baseui-menu.js`/`composite.js`/`floating.js` to React `floating-ui-react`/`composite` — followed by maintainer synthesis, white-box code verification of every candidate gap, bUnit, in-app browser inspection of the Blazor docs demos, Playwright (Server + WASM), and PNPM source-doc validation against the live React docs.

Per the shared-utility lesson recorded in prior audits, the upstream delta was diffed against the **shared** floating/composite infrastructure, not only the `menu/` directory. Every claimed gap was re-verified against the React source before any change — three sub-agent claims were **rejected as false positives** during verification (see Rejected Findings).

## Baseline

The Blazor Menu port's behavioral content predates the Dialog/Popover audits: the only Menu commits in the Blazor repo are `7b77fb9d` (rename) and `15d9a824` (internal-components move); the substantive port and its `.audit` artifacts date to mid-May 2026. The shared `blazix-baseui-floating.js` was subsequently brought current by the Popover (`2d815ba0`) and Dialog (`5e2b4046`) audits, so Menu **inherits** the recent shared focus-manager fixes (confirmed below). The delta surface evaluated was therefore every upstream `[menu]`, `[composite]`, `[popups]`, and shared `[floating]` change from ~2026-02 through HEAD `748f4228d`.

| 1 | `#4826` group labels in radio groups | `Menu/MenuRadioGroup.razor` (+ `MenuRadioGroupTests` / contract) | `MenuRadioGroup` now wraps its children in `CascadingValue<IMenuGroupContext>` exposing `SetLabelId`, tracks `labelId`, and emits `aria-labelledby={labelId}` (user-supplied `aria-labelledby` via `AdditionalAttributes` overrides). | 4 | `#4195` typeahead matches CSS-hidden items | `wwwroot/blazix-baseui-menu.js` (+ regenerated `.min.js`) | Added `isMenuItemVisible` (mirrors React `isElementVisible`: `isConnected` + `checkVisibility()` / computed `display`/`visibility`) and applied it in both typeahead match loops and the no-match `hasMatch` check. | 5 | `MenuTypedTrigger` interaction type | `Menu/MenuTypedTrigger.razor` | Added the missing `RootContext?.SetInteractionType?.Invoke("keyboard")` in the ArrowDown/ArrowUp open branch, matching `MenuTrigger`, so keyboard-opened typed-payload menus report the correct interaction type (drives focus-visible on open). |
| 6 | Multi-trigger switch dismissed as outside-press | `Menu/MenuRoot.razor`, `wwwroot/blazix-baseui-menu.js` (+ `.min.js`), Playwright `MenuTestsBase.cs` | **Bug:** in controlled / multiple-trigger / animating-viewport menus, clicking a *sibling* trigger to switch the open menu made it "briefly open then close." Root cause: `handleGlobalPointerDown` excluded only the **active** `rootState.triggerElement`, so a press on any other registered trigger satisfied `!clickedInsidePopup && !clickedOnTrigger` → `OnOutsidePress` → close, racing the switch. All repairs are additive and surgical. The 216 pre-existing Menu bUnit tests are unaffected; two new bUnit tests (`MenuRadioGroupTests.GroupLabelWiresAriaLabelledby`, `MenuPopupTests.PopupIsLabelledByTrigger`) and two new Playwright tests (`MultiTrigger_Contained_SwitchWhileOpen`, `MultiTrigger_Handle_SwitchWhileOpen`) were added and pass. ## Residual Risk

Tracked non-behavioral / deferred items, each with the exact upstream mechanism and the reason it is accounted-for rather than patched:

- **`#4231`/`#4723` safePolygon `blockPointerEvents` scoping (deferred refinement).** React applies `applySafePolygonPointerEventsMutation` (`useHoverInteractionSharedState.ts:87`) from **two** hooks — `useHoverReferenceInteraction` (during close-traversal) and `useHoverFloatingInteraction` (for the open duration) — resolving the scope element as `parentFloating ?? cachedScope ?? ref.closest('[data-rootownerid]') ?? doc.body` with a guard that the scope is not the floating element itself (`#4723` keep-mounted), tracking ownership in a `WeakMap`, setting `scope{pointer-events:none}` + `reference/floating{pointer-events:auto}`. The Blazor `createHoverInteraction` threads `blockPointerEvents` to `handleClose.__options` but applies no mutation. The core safePolygon close-delay triangle IS implemented and submenu hover is functional (validated). This refinement is deferred because a faithful port mutates shared `blazix-baseui-floating.js` (used by tooltip/preview-card/popover hover) on a fragile, heavily-iterated hover system, and cannot be reliably validated without extensive cross-component pointer testing. It is gated per-call (`blockPointerEvents` is only `true` for menu hover), so a future contained port would not affect other components.
- **`#4990` submenu hover rest-delay (approximation).** Blazor uses an `allowMouseEnter`-gated delay swap rather than React's `restMs` + `shouldOpen` veto. The configured open delay is honored once the parent has seen movement; the divergence is the exact "pointer must rest without moving" timing model.
- **`#4604` clear-highlight on scroll-clipped pointer-leave.** The common pointer-leave clear is covered by each item's C# `HandleMouseLeave`; the variant where an item scrolls under a stationary pointer inside a scroll container is not specially handled.
- **`#4891` `MenuSubmenuRoot` exposes `Handle`/`TriggerId`/`DefaultTriggerId`.** React removed these from `SubmenuRoot`. In Blazor they are forwarded to the inner root but inert (nested menus suppress modal/handle logic). Removing public parameters is a breaking change deferred to an API pass.
- **`MenuRadioItemIndicator` robustness divergence.** The Blazor `MenuRadioItemIndicator` lacks the `transitionVersion`/stale-callback/deferred-transition machinery that `MenuCheckboxItemIndicator` has (React's two indicators are identical). Both are functionally correct; the radio indicator is less hardened against rapid toggles mid-exit-animation.
- **`#5c7ad60dd`/`#4922` menubar fixes** target the separate **MenuBar** component; flagged for a MenuBar audit.

The unavoidable Blazor event-model difference persists: consumers cancel Base UI behavior through `OnOpenChange` → `args.Cancel()` / `OnValueChange` / `OnCheckedChange` rather than mutating a browser event, and JS-driven dismissal (escape/outside/focus-out) is handled in `blazix-baseui-menu.js` rather than React's `useDismiss`.
